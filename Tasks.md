# Tasks

## Open — High Priority

- [ ] **Footer binary bytes leaking into output** — ~96 files. After the human-readable document reference text in the footer (e.g. `CND 6.0,  10 May 2018`), binary layout/format bytes that follow in the footer zone are parsed as content and appended to the footer paragraph. Pattern observed: `[PAGE]CND 6.0,  10 May 2018?0??X@xxd A` where `?` = unmapped high bytes and `@xxd` (`40 78 78 64`) = printable ASCII from a binary parameter block. Also a spurious leading `3 . ` or `2 . ` prefix on some footer lines may indicate a page-counter or per-section byte leaking into the footer zone prefix. Root cause is a footer zone termination problem — the parser does not stop consuming footer text at the correct point. Investigation documented in `Investigation.md`. Starting file: ROBBS.3.

## Open — Medium Priority

- [ ] **`??` at start of documents** — 39 files show a spurious `??` first paragraph before the actual content. Most are in the `22 66 0b` group and may resolve once the mixed ctrl_byte fix is in place (the `??` could come from unmapped high bytes in an unrecognised `22 61 0b` block at the document start). Re-evaluate after the mixed ctrl_byte task is complete.

## Open — Low Priority

- [ ] **`==` artifact in ABBEYGRD.4 / ABBEYGRD.BAK** — Two files showing `"==` and trailing `=` in war-memorial sections. Same PASSCHENDAELE compound variant (`22 3d 3d`) that was fixed in MEMORIAL.002 via a 10-byte skip + doubled-pair strip in `_skip_ctrl_sequence`. ABBEYGRD.4 is a `22 61` file — verify the existing PASSCHENDAELE guard fires correctly for this file and the doubled pair is consumed. May be a quick fix or no-op if the guard already covers it.

## Known Limitations (longer-term / future work)

- [ ] **JOY format parser** — 233 JOY files in the sample set cannot currently be converted. Two sub-versions exist: `01 04` (word sep `0x0a`, para break `07 02 0a 55 0e XX YY 06 ZZ ZZ`) and `01 02` (word sep `0x01`, different structure). Implementing proper support would require a separate parser alongside the existing DOC parser.

- [ ] **Untested document types** — Parser was developed against a single sample file. Locoscript 2 letters, labels, and other document types may surface unrecognised control sequences.

- [ ] **Page margin byte investigation** — Bytes at `0x2BA`/`0x2BC` in the file header appear to encode page margins, but their unit system is unconfirmed (HENCOTES=`0x28`, BREWERS.5=`0x30`). Once decoded, the footer tab stop positions (currently hardcoded to A4 1-inch margins) can be derived dynamically from the file header. See `Page-Margin-Investigation.md` (local only) and Requirements.md known limitation.

- [ ] **Right alignment control sequence investigation** — `10 07` appears 5016 times across 393 sample files. The most common byte pattern immediately after is `1d 66 XX 01 YY YY` (a possible 6-byte control block not yet handled), followed by `03 00 00 00 00 00` and various `22 XX YY` forms. The file encodings reference suggests `10 07` may consume additional parameter bytes. Investigation needed: identify what the `1d 66 XX` prefix represents, whether it is always present after `10 07`, and whether the current 2-byte skip leaves parameter bytes that should be consumed, potentially causing artefacts in right-aligned paragraphs.

- [ ] **Template / binary header documents** — A small number of files (e.g. SLUMHOLY.ISL, A4LETTER) appear to be LocoScript document templates rather than content documents. The first paragraph contains template metadata (`Template for chapters of a book ... PCW9512England`) followed by binary layout data. These are a different document type and not a parser bug — no fix required unless there is a need to identify and skip them gracefully.
