# Tasks

## Known Limitations (future work)

- [x] **Output filename collision on multi-part extensions** — Fixed in branch `feature/output-filename-collision`. Changed `src.with_suffix(fmt)` to `src.parent / (src.name + fmt)` in `app.py` so the new extension is always appended (`BUILDNGS.A-C` → `BUILDNGS.A-C.docx`) rather than replacing the existing one. 67/67 tests passing.

- [x] **Variable control prefix (`22 6d` vs `22 61`)** — Fixed in branch `feature/variable-ctrl-prefix`. Added `_detect_ctrl_byte()` which scans for the first `22 XX 0b` occurrence and returns `XX` (defaults to `0x61`). Threaded the detected byte through `_find_content_start`, `_skip_ctrl_sequence`, and the main parse loop. Files using `22 6d 0b` (e.g. `BUILDNGS.A-C`, `MARKETPL.*`) now produce correct content. 6 new tests. 73/73 tests passing.

- [x] **Running header/footer extraction** — Implemented in branch `feature/header-footer-extraction`. Added `_find_body_start` (c4 0e method + B5≠`0x14` fallback scan) and `_find_footer_start` (scans for section break between `first_para` and `body_start`). Position-based section routing at `22 XX 0b` blocks replaces the unreliable `_section_type_at` calls inside the parse loop. Added pre-body `00 00` NUL flush, B3=`0x00` layout-block skip, and `01 XX XX` 3-byte trailing indent fallback. Also fixed variable ctrl sequence skip (`i += 3` instead of `i += 4`) to prevent `0x13` formatting-prefix bytes being consumed before the main loop can recognise them. Footer page-number zone (centre-aligned `==` placeholder + page token machinery) is silently discarded; only the right-aligned document reference text is extracted. Confirmed on HENCOTES (header + footer) and BUILDNGS.H (footer only). 13 new tests. 89/89 tests passing.

- [ ] **Tab stop position preservation** — The layout table at `0x2C6` in the file header contains custom tab stop positions (scale pitch units; confirmed in Requirements.md). Currently these are not read, so tab characters (`\t`) in the output fire at the default tab stop position rather than the author's intended column. Visible in HENCOTES where a `09 05 01 1c 1c` tab between "The Place Names of Hexham Region" and ", Vic Watts," produces an oversized gap. Investigation complete — see [`Tab Stop Investigation.md`](Tab%20Stop%20Investigation.md) for full findings and implementation plan. Key findings: (1) `09 05 01 XX XX` param bytes encode the explicit tab column in scale pitch units; (2) `0f 04 B1 B2` B1 byte encodes the indent/tab column, B2 is always the body ctrl_byte; (3) layout table stores the named stop positions and the scale pitch conversion factor; (4) `_detect_ctrl_byte()` has a latent bug for files like Memorial.002 where the header zone uses a different ctrl_byte from the body (fix: use most-frequent, not first).

- [ ] **Contents Page control block investigation** — In `22 6d` files (e.g. `BUILDNGS.A-C`), the Contents Page paragraph is being produced as a single run rather than one paragraph per entry. Two symptoms observed: (1) a spurious `x` character at the start of the paragraph; (2) `"m` separators between entry names instead of paragraph breaks (suggesting `22 6d` sequences followed by a word separator are being emitted as literal text rather than being consumed as paragraph delimiters). Needs hex dump analysis to identify the exact control structure used between Contents Page entries and determine the correct fix.

- [ ] **Conversion output size warning** — After converting a file, if the output is less than 10% of the size of the input file (in bytes), log a warning in the error log and surface it to the user. Intended to catch cases where the parser has silently failed to extract most of the content (e.g. unrecognised control prefix). Threshold of 10% chosen to account for binary header/control overhead in source files. Add as a new Conversion Behaviour requirement in Requirements.md when picking this up.

- [ ] **Section break creates spurious paragraph split in flowing text** — `0e 01`/`0e 02` section/page breaks currently always flush the paragraph and start a new one. In some files (e.g. `BUILDNGS.H`) this splits continuous prose mid-sentence — e.g. "...for the elderly, " / "opened by the chairman..." — because the break represents a layout change (likely a column switch; the layout block following the `0e 01` contains a value of `0x02` consistent with a 2-column layout descriptor) rather than a paragraph boundary. Needs investigation to determine whether the break should suppress paragraph creation when text flows continuously across it.

- [ ] **JOY format parser** — 233 JOY files in the sample set cannot currently be converted. Two sub-versions exist: `01 04` (word sep `0x0a`, para break `07 02 0a 55 0e XX YY 06 ZZ ZZ`) and `01 02` (word sep `0x01`, different structure). Implementing proper support would require a separate parser alongside the existing DOC parser.

- [ ] **Untested document types** — Parser was developed against a single sample file. Locoscript 2 letters, labels, and other document types may surface unrecognised control sequences.

- [ ] **Font switching investigation** — The font table at `0x138` (10 × 28-byte entries) is confirmed, with font names ("CG Times", "LX Roman", "Courier Condensed", etc.), but font references in the text stream have not been identified. The colleague's table notes `08 0b` as "function unknown (in A4LETTER)" which may be font-related. Investigation needed: find files with known font changes, hex dump around the switch points, and identify the control sequence(s) responsible. Implement font mapping to RTF (`\f0`, `\f1` etc.) and DOCX (`run.font.name`) once confirmed.

- [ ] **Shadow Copy mode** — New application mode to mirror-convert an entire folder structure. See Requirements.md for full spec. Do this last.

- [x] **Paragraph alignment (centre/right)** — Implemented in branch `feature/paragraph-alignment`. Investigation confirmed: `11 06` (DC1 + ACK) = centre alignment; `10 07` / `10 04` (DLE + BEL/EOT) = right alignment. Both appear immediately after `22 61 0b` paragraph content blocks in 4,257 and 4,422 occurrences respectively across the sample set. Added `alignment` field to `Paragraph`; 2-byte handlers in `parse()` (the `06`/`07`/`04` param byte is now consumed rather than emitting a spurious space). RTF uses `\qc`/`\qr` on `\pard`. DOCX uses `WD_ALIGN_PARAGRAPH`. Golden fixture regenerated. 67/67 tests passing.
- [x] **Tab handling in converter output** — Fixed in branch `feature/tab-handling`. TXT now uses `strip(' \n')` (preserving tabs at paragraph edges). RTF maps `\t` → `\tab ` control word in `_rtf_escape` and no longer strips run text before escaping. DOCX no longer strips run text and removes the spurious trailing space appended to every run. 45/45 tests passing.
- [x] **Remove "no file extension" restriction** — Updated Requirements.md: input files may have any extension or none. Shadow Copy mode spec updated to identify files by `DOC` magic bytes rather than absent extension. No code changes needed — the app already handles both cases correctly.
- [x] **Detailed file header map** — Investigated. Summary field (0x05–0x5E) contains template descriptions, not document titles — not useful. Version bytes always `01 03` — not useful. Font table at `0x138` (10 × 28 bytes) confirmed with font names ("CG Times", "LX Roman" etc.) but font references in the text stream have not been identified, so font mapping is blocked for now. Layout table at `0x2C6` (10 × 73 bytes) confirmed: scale pitch at entry offset +11 (`0x18` = 10cpi = 0.1 inch/unit), point size at +13 (pt × 10, `0x78` = 12pt), 15 tab stop positions at +33–+47 (scale pitch units; `0x18` = default 2.4", custom values confirmed). Document-level margins appear around `0x2B8` (two 16-bit fields, `0x28` or `0x30` across files) but unit system unconfirmed — left unresolved. Tab stop extraction feeds into the tab handling task. First-paragraph junk unaffected.
